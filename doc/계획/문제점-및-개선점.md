## 1. CRITICAL — B 키 선택 후 강제종료 시 선택 해제

| 항목 | 내용 |
|---|---|
| **검증** | **확인됨** + **보완됨** |
| **파일** | `LOGIC/LoadImage.cs` `SelectEx` ~626–656, `SelectedUpdate` ~514–561 · `MainWindow.xaml.cs` `Window_Closing` ~279–297 · `LOGIC/SelectorSetting.cs` |
| **묶음** | **1** |

### 현상

`B` 키(또는 체크박스)로 사진을 선택한 뒤 프로세스가 강제 종료되면, 재시작 시 선택이 사라져 있다.  
정상 종료 직전 마지막 몇 번 선택도 사라질 수 있다.

### 근본 원인 (3가지 복합)

#### 원인 1: `SelectEx()` fire-and-forget 저장 — 확인됨

```csharp
// LoadImage.SelectEx()
_ = Common.NowSelectorSetting.SelectedSet.Contains(NowImage.FileName)
    ? Common.NowSelectorSetting.SelectedSet.Remove(NowImage.FileName)
    : Common.NowSelectorSetting.SelectedSet.Add(NowImage.FileName);

Common.NowSelectorSettingDate = File.GetLastWriteTime(settingPath); // 저장 전 시각
SelectedUpdate();
UpdateCount();
Task.Run(() =>  // 대기 없음
{
    try { Common.NowSelectorSetting.Save(); }
    catch (Exception ex) { Log.Exception(ex); }
});
```

- 메모리 `SelectedSet`만 먼저 바꾸고, 디스크 쓰기는 `Task.Run`에 던진다.
- 강제 종료 시 YAML에 반영되지 않는다.

#### 원인 2: `Window_Closing`에서 SelectorSetting 미저장 — 확인됨

```csharp
// MainWindow.Window_Closing
App.Setting.Save();                 // ApplicationSetting만
Application.Current.Shutdown();     // SelectorSetting.Save() 없음
```

- 정상 종료 시에도 폴더 선택 YAML을 명시 저장하지 않는다.
- `Shutdown()`은 미완료 `Task.Run` 저장을 보장하지 않는다.

#### 원인 3: `NowSelectorSettingDate`를 저장 **전**에 갱신 → 재로드 경합 — 확인됨 + 보완

`SelectedUpdate()` (이미지 이동·표시 시 호출):

```csharp
if (Common.NowSelectorSetting == null
    || File.GetLastWriteTime(settingPath) != Common.NowSelectorSettingDate)
{
    Common.NowSelectorSetting = SelectorSetting.Load(NowDir); // 디스크 기준으로 메모리 덮어씀
    Common.NowSelectorSettingDate = File.GetLastWriteTime(settingPath);
}
```

시나리오:

1. B 키 → 메모리에 선택 반영 → `NowSelectorSettingDate` = **저장 전** mtime  
2. `Task.Run(Save)` 아직 미완료  
3. 화살표 이동 → `SetImage` → `SelectedUpdate`  
4. 디스크 mtime과 `NowSelectorSettingDate` 비교  
   - 저장이 **끝난 뒤**: mtime 변경 → 디스크에서 재로드 → 저장된 선택이 보이면 정상  
   - 저장이 **끝나기 전**: mtime 불일치가 아니거나, 빈/이전 YAML을 로드 → **방금 메모리 선택이 날아감**

보완: yaml이 처음 없을 때 `SetArg`는 `NowSelectorSetting = null`이고, 첫 `SelectEx`가 `SelectorSetting.Load`로 **빈 파일을 생성**한 뒤 비동기 저장한다. 강제 종료 시 “파일은 있는데 선택 목록은 비어 있는” 상태가 된다.

### 수정 방안 (권장: A + B)

#### A. `SelectEx` 동기 저장 + 저장 후 mtime 갱신

```csharp
try
{
    Common.NowSelectorSetting.Save();
    Common.NowSelectorSettingDate = File.GetLastWriteTime(settingPath); // 저장 후
}
catch (Exception ex)
{
    Log.Exception(ex);
}
```

- YAML이 작아 UI 스레드 동기 I/O 부담은 실측상 미미한 편이다.

#### B. `Window_Closing`에서 최종 저장

```csharp
if (Common.NowSelectorSetting != null)
    Common.NowSelectorSetting.Save();
```

DLL 삭제·`App.Setting.Save()` 전에 호출.

### 수락 기준

- [ ] B 키 10회 선택 → 정상 종료 → 재실행 → 10개 유지  
- [ ] B 키 선택 직후 작업 관리자 강제 종료 → 재실행 → 가능한 한 마지막 저장분 유지(동기 저장 적용 시 직전 선택까지 유지)  
- [ ] 선택 직후 화살표 연타 시 선택이 풀리지 않음

## 2. CRITICAL — NearLoad 프리패치가 잘못된 인덱스 사용

| 항목 | 내용 |
|---|---|
| **검증** | **확인됨** |
| **파일** | `LOGIC/LoadImage.cs` `NearLoad` ~151–186, 특히 **164행** |
| **묶음** | **2** |

### 문제

```csharp
int index = NowIndex + (j == 0 ? i * 1 : i * -1);  // 이웃 인덱스 계산은 맞음
if (index < 0 || index > LastIndex) { continue; }
ImageData data = ImageList.ElementAt(i).Value;       // BUG: i 사용 (0~9만)
```

- `index`로 bounds 검사만 하고, 실제 로드 대상은 루프 변수 `i`(0..9)다.
- 결과: **항상 리스트 앞쪽 0~9번**만 프리패치. 현재 위치 ±이웃은 캐시되지 않는다.
- 부가: `i==0`일 때 `j=0/1` 모두 `index == NowIndex`라 현재 항목을 두 번 검사한다(의도는 ±방향 루프).

### 수정

```csharp
ImageData data = ImageList.ElementAt(index).Value;  // i → index
```

### 수락 기준

- [ ] 폴더 중간(예: 50번째)에서 좌우 이동 시 이웃 썸네일이 미리 준비되어 체감 지연 감소

## 3. CRITICAL — MetaProvider.GetSize 네이티브 메모리 누수

| 항목 | 내용 |
|---|---|
| **검증** | **확인됨** + **보완됨** |
| **파일** | `LOGIC/MetaProvider.cs` `GetSize` ~77–97 · 호출: `MANAGER/RawManager.cs` 16, 61행 |
| **묶음** | **3** |

### 문제

```csharp
IntPtr handler = RAWLib.libraw_init(...);
RAWLib.libraw_open_file(handler, _path);   // 반환 에러 코드 무시
var ptr = RAWLib.libraw_dcraw_make_mem_image(handler, ref errc);
var img = PtrToStructure<...>(ptr);
// libraw_dcraw_clear_mem(ptr) 없음
// libraw_close(handler) 없음
```

- 폴백 경로마다 네이티브 핸들·이미지 버퍼 누수.
- `open_file`/`make_mem_image` 실패 시 `ptr == IntPtr.Zero`에서 마샬링 시 **AV 가능**.

### 보완 (발생 빈도)

- `GetSize()`는 **모든 RAW 디코드**에서 호출된다 (`RawManager.GetThumbnailArray`).
- 다만 EXIF에 `_.Size.Width` / `_.Size.Height`가 있으면 **조기 return** → libraw 폴백·누수 없음.
- 해당 키가 없는 카메라/파일에서만 폴더 단위로 누수가 쌓인다. “모든 RAW마다 무조건”은 아님.

### 추가 위험 (보완)

- 폴백이 `libraw_open_file`(ANSI)만 사용. 한글 등 비 ANSI 경로에서는 `libraw_open_wfile`이 더 안전하다.

### 수정 요지

```csharp
IntPtr handler = IntPtr.Zero;
IntPtr ptr = IntPtr.Zero;
try
{
    handler = RAWLib.libraw_init(...);
    if (RAWLib.libraw_open_file(handler, _path) != RAWLib.LibRaw_errors.LIBRAW_SUCCESS)
        return new Size(800, 600);
    // 가능하면 open_wfile 검토

    var errc = 0;
    ptr = RAWLib.libraw_dcraw_make_mem_image(handler, ref errc);
    if (ptr == IntPtr.Zero) return new Size(800, 600);

    var img = PtrToStructure<RAWLib.libraw_processed_image_t>(ptr);
    ...
}
finally
{
    if (ptr != IntPtr.Zero) RAWLib.libraw_dcraw_clear_mem(ptr);
    if (handler != IntPtr.Zero) RAWLib.libraw_close(handler);
}
```

### 수락 기준

- [ ] EXIF 사이즈 없는 RAW 다수 연속 로드 시 프로세스 핸들/메모리 급증 없음  
- [ ] 손상/미지원 파일에서 AV 없이 기본 Size(800,600) 등으로 폴백

## 4. HIGH — 선택 내보내기/복사 시 NullReferenceException

| 항목 | 내용 |
|---|---|
| **검증** | **확인됨** + **수정됨**(단락 회로 설명) |
| **파일** | `VIEW/ExportThumbnailWindow.xaml.cs` ~171 · `VIEW/RawCopyWindow.xaml.cs` ~183 |
| **묶음** | **2** |

### 문제

```csharp
if (Common.NowSelectorSetting == null && Common.NowSelectorSetting.SelectedSet.Count == 0)
```

**이전 문서 오류 수정:** C#의 `&&`/`||`는 **단락 평가**를 한다. “양쪽을 항상 평가”가 아니다.

실제 NRE 경로:

| `NowSelectorSetting` | `SelectedSet` | 동작 |
|---|---|---|
| `null` | (해당 없음) | 좌변 `true` → 우변 평가 → **NRE** |
| non-null, Count==0 | 비어 있음 | 좌변 `false` → 단락 → 조건 전체 false → **조기 return 안 함** → 빈 루프 후 “완료” 메시지 |
| non-null, Count>0 | 있음 | 정상 진행 |

의도(“설정 없거나 선택 없으면 중단”)에는 `||`가 맞다.

```csharp
if (Common.NowSelectorSetting == null || Common.NowSelectorSetting.SelectedSet.Count == 0)
```

`SetArg`에서 yaml 미존재 시 `NowSelectorSetting = null`이므로, 해당 폴더에서 “선택한 사진만” 내보내기/복사 시 재현 가능하다.

### 수락 기준

- [ ] yaml 없는 폴더에서 선택 내보내기/복사 → NRE 없이 “선택한 사진이 없습니다.”  
- [ ] 선택 0개일 때도 동일 안내

## 6. HIGH — processingSet 누수로 이미지 영구 미로드

| 항목 | 내용 |
|---|---|
| **검증** | **확인됨** + **보완됨**(경로 재정리) |
| **파일** | `LOGIC/LoadImage.cs` `GetThum` ~235–278, `AllLoad` ~188–234, `NearLoad` ~151–186 |
| **묶음** | **3** |

### 문제 경로 A — `GetThum` early-return (기존 문서)

```csharp
if (IsSetImage && NowIndex != data.Index)
{
    return;  // TryRemove 없음
}
// ...
processingSet.TryRemove(data.Path, out _);  // 정상 경로만
```

**보완:** 네비게이션(`PrevImage`/`NextIamge`)은 `GetThum(data, true)` 전에 **`processingSet.TryAdd`를 하지 않는다.**  
early-return만으로 누수되려면, 그 전에 `NearLoad`/`AllLoad`가 같은 경로를 `TryAdd`한 상태여야 한다.

실전 시나리오:

1. `NearLoad`가 이웃 경로를 `TryAdd` 후 `GetThum(data, false)` 시작  
2. 사용자가 그 사진으로 이동 → `GetThum(data, true)` 추가 기동  
3. 사용자가 바로 다른 사진으로 이동 → `GetThum(true)` early-return (**Remove 안 함**)  
4. `GetThum(false)`가 예외 등으로 Remove에 실패하면 해당 키 **영구 잔류**  
5. 이후 `NearLoad`/`AllLoad`는 `ContainsKey`로 **영원히 스킵**

예외 시 `TryRemove`가 없는 것도 동일 계열 버그 → **`try/finally`가 정석**.

### 문제 경로 B — `AllLoad` else 누락 (신규)

```csharp
if (!processingSet.ContainsKey(f.Key))
{
    processingSet.TryAdd(f.Key, null);
    if (!f.Value.IsNotImage && f.Value.ImageArray == null)
        GetThumNotThread(f.Value);  // 내부에서 Remove
    // else: 이미 로드됨/IsNotImage → TryAdd만 하고 Remove 없음 → 키 잔류
}
```

`NearLoad`는 else에서 `TryRemove`를 한다. `AllLoad`는 하지 않는다.  
이미 로드된 항목은 `ImageArray != null`이라 당장 안 보일 수 있으나, **실패 후 재시도·상태 전이**를 막는다.

### 수정

```csharp
// GetThum 전체
try
{
    Thread.Sleep(10);
    if (IsSetImage && NowIndex != data.Index) return;
    // 디코드 ...
}
finally
{
    processingSet.TryRemove(data.Path, out _);
}

// AllLoad: TryAdd 후 디코드 안 하는 분기에서도 TryRemove
```

### 수락 기준

- [ ] 빠른 좌우 이동 + 백그라운드 프리패치 후에도 미로드 이미지가 나중에라도 로드됨  
- [ ] 동일 폴더 재오픈 없이 실패 파일이 영구 스킵되지 않음(정책에 따라 재시도)

### 이전 문서 대비 주요 정정

1. **`&&` “양쪽 항상 평가”** → C#은 단락 평가. NRE는 “좌변이 true일 때 우변 평가”로 설명 수정.  
2. **GetSize 누수 빈도** → 모든 RAW가 아니라 EXIF 사이즈 키 부재 시에만 libraw 폴백.  
3. **processingSet** → navigation 단독 early-return만으로는 부족; NearLoad 경합 + **AllLoad else 미Remove** 추가.  
4. **AllLoad** → 100% 초과뿐 아니라 **1장 폴더 0 나눗셈** 명시.  
5. **Costura unmanaged** → 현 구조와 단순 치환 불가, spike/대안 필요.  
6. **신규** → 프로세스/OS 비트 불일치, `libraw_open_wfile` 권장, AllLoad processingSet.

### 권장 진행 방식

한 묶음이 끝날 때마다:

1. 해당 묶음 수락 기준 수동 확인  
2. (가능하면) Debug 빌드  
3. 이 문서 해당 항목에 `✅ 완료` 표시 후 다음 묶음  

바로 시작 시: **묶음 1 → 묶음 2 → 묶음 3** 순이 투자 대비 효과가 가장 크다.
