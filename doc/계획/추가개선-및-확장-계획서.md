# FastRawSelector 추가 개선 및 확장 계획서

> 작성일: 2026-07-15  
> 전제: 안정성 묶음 1~8 및 AppData 통합이 반영된 이후 단계  
> 관련 문서: `doc/계획/문제점-및-개선점.md`, `src/FastRawSelector/TODO.txt`

---

## 1. 목적

버그·데이터 손실·네이티브/경로 이슈 등 **큰 문제점 대응은 일단락**된 상태로 본다.  
이후 목표는 다음 세 축을 균형 있게 진행하는 것이다.

| 축 | 목표 |
|---|---|
| **품질** | 주석·로그·진단성으로 “고칠 수 있는 코드” 유지 |
| **완성** | 미루어 둔 스텁(설정·그리드·분류·EXIF 변경 등)을 실제로 쓸 수 있게 |
| **확장** | 속도·편의·배포·다국어 등 제품 수준 기능 |

이 계획서는 **무엇을 어떤 순서로** 할지 고정하고, 한 번에 범위를 키우지 않기 위한 로드맵이다.

## 2. 현재 상태 요약

### 2.1 완료로 보는 영역 (1단계)

- 선택 저장 동기화 / 종료 시 저장
- NearLoad 인덱스, AllLoad 진행률, processingSet 누수
- MetaProvider.GetSize 네이티브 해제
- 내보내기/복사 NRE, 예외 로그(디코드 경로), PostBuild config 유지
- SelectorSetting 경로 인스턴스화, 최소 동기화
- `IsWindowReady`, 프로세스 비트 기준 DLL 추출
- AppData 통합: `%AppData%\Roaming\FastRawSelector\` (설정·로그·네이티브)
- `AllowNotRawImage` 복원, `NextImage` 리네임, 일부 데드 코드 정리

### 2.2 아직 미완·미구현

### 2.3 의도적으로 유지하는 제약

- **.NET Framework 4.6.2 + WPF**, Windows 전용
- **코드비하인드 + static 서비스** 스타일 유지 (전면 MVVM 전환은 범위 밖)
- UI 문자열 **한국어 기본** (다국어는 후순위 단계)
- 테스트 프로젝트 없음 → 단계마다 **수동 검증 시나리오** 필수

## 3. 원칙 (개발 규칙)

기존 `AGENTS.md` 가이드와 동일하게 적용한다.

1. **Think before coding** — 모호하면 가정 명시, 여러 해석 제시  
2. **Simplicity first** — 요청 범위 밖 추상화·설정 남발 금지  
3. **Surgical changes** — 한 단계 = 한 목표, 인접 리팩터 금지  
4. **Goal-driven** — 수락 기준을 단계마다 적고, 빌드·수동 확인까지 완료  

추가 원칙:

- **로그 먼저, 기능 나중** 하지 않는다. 로그/주석은 **기능 단계와 병행**하되, 3단계(관측성)를 기능 대공사 앞에 짧게 둔다.  
- 스텁 창을 **삭제하지 말고 구현**한다 (이미 UI 골격이 있음).  
- 사진 폴더의 `FastRawSelector.yaml`(선택 상태)과 AppData 설정은 **역할을 분리** 유지.

## 4. 전체 로드맵 (단계)

```
[완료] 1단계 안정화 묶음 1~8 + AppData
        ↓
[다음] 2단계 관측성·문서화 (주석 + 로그 체계)
        ↓
[다음] 3단계 설정·사용성 (설정창, 도움말, Always on top 등)
        ↓
[다음] 4단계 미완 기능 완성 (그리드 / 폴더분류 / EXIF변경)
        ↓
[다음] 5단계 성능·파이프라인 (Exif 지연, 프리패치 고도화)
        ↓
[선택] 6단계 배포·확장 (설치, 연결 프로그램, 다국어)
```

각 단계는 **에픽 단위**로 끊고, 에픽 안은 번호 붙은 작업 묶음으로 진행한다.

## 5. 2단계 — 관측성·문서화 (최우선 권장)

**목표:** 다음에 기능을 추가할 때 원인 추적과 온보딩이 가능하도록 기반을 깐다.  
**규모:** 소~중 · **의존:** 없음 · **기간 감:** 짧음

### 5.1 로그 정책

| 레벨 | 언제 쓸지 | 예시 |
|---|---|---|
| **INFO** | 사용자 의미 있는 수명주기 | 기동, 폴더 오픈, 선택 저장, 내보내기/복사 시작·완료, 종료 |
| **DEBUG** | 개발용 상세 (기본 OFF 가능) | NearLoad 범위, 디코드 성공 경로, 프리패치 스킵 사유 |
| **WARN** | 복구 가능한 이상 | 설정 마이그레이션 실패 후 기본값, 파일 일부 스킵 |
| **ERROR** | 실패·예외 (이미 일부 존재) | 디코드 실패, 저장 실패, 네이티브 초기화 실패 |

**출력 위치 (이미 구축됨):**

`%AppData%\Roaming\FastRawSelector\logs\`

**추가 작업:**

| ID | 작업 | 파일 후보 | 수락 기준 |
|---|---|---|---|
| L-1 | 기동/종료 INFO | `App.xaml.cs`, `MainWindow` Closing | logs\INFO 에 기동·종료 1줄 이상 |
| L-2 | 폴더 로드 INFO | `LoadImage.SetArg` | 경로, 파일 수, AllowNotRawImage 여부 |
| L-3 | 선택 토글/전체해제 DEBUG 또는 INFO | `SelectEx`, `AllDeselect` | 파일명·선택 수 (연타 시 과다하면 DEBUG) |
| L-4 | 내보내기/복사 시작·완료·실패 건수 | Export / RawCopy | 대상 경로, 성공/실패 카운트 |
| L-5 | 네이티브 초기화 결과 | `InitNativeRuntime` | AppDataPath, NativePath, 64/32 |
| L-6 | 설정에 로그 레벨 | `ApplicationSetting` + 설정창(3단계) 또는 YAML | 기본 INFO, DEBUG 전환 가능 |
| L-7 | 루트 레벨과 appender 정리 | `log4net.dat` | INFO 기본 시 DEBUG 파일 비대화 정책 명확화 |

**로그 안티패턴 (하지 말 것):**

- 매 프레임/매 키 입력 INFO  
- 비밀번호·전체 경로 과다 나열 금지 수준은 개인 PC 앱이라 완화하되, **수천 줄 스팸**은 금지  
- catch에서 `Log.Exception` 없이 삼키기 (이미 줄인 패턴 유지)

### 5.2 주석 정책

| 종류 | 규칙 |
|---|---|
| **파일/클래스 머리** | 역할 2~5줄 (한국어 OK, 기존 혼용 유지) |
| **공개 메서드** | “무엇을/언제/부작용(전역 상태)” 한 블록 |
| **왜(Why)** | 비자명한 결정만 (예: 동기 저장 이유, SetDllDirectory 이유) |
| **무엇(What)** | 코드가 말하는 내용 반복 금지 |
| **TODO** | `// TODO(단계ID):` 형식으로 계획서 ID 연결 |

**우선 주석 대상 파일 (중요도 순):**

1. `LOGIC/LoadImage.cs` — 파이프라인·프리패치·선택  
2. `LOGIC/Common.cs` — AppData, Invoke, 파일 판별  
3. `MANAGER/RawManager.cs` / `BitmapManager.cs`  
4. `LOGIC/MetaProvider.cs` / `RAWLib.cs` (P/Invoke 주의점만)  
5. `MainWindow.xaml.cs` — 단축키 맵  

**작업 ID: C-1 ~ C-5** — 위 파일 순서로 “머리 + 핵심 public API”만. 전 파일 주석 달기는 하지 않는다.

### 바로 다음에 할 일 (제안)

```
묶음 A (로그 수명주기 + LoadImage/Common 주석)
  → 묶음 B (로그 레벨 + 나머지 핵심 주석)
  → 묶음 C (설정창 + 도움말)
  → 묶음 E (폴더 분류) …
```

기능 욕심이 크면 **A → C → E** 로 관측성 최소만 하고 기능을 당겨도 된다.  
다만 **A 없이 대규모 기능**을 넣으면 회귀 추적이 다시 어려워진다.

## 12. 아키텍처 확장 시 가이드 (과하지 않게)

기능을 늘릴 때 **당장 MVVM으로 갈아엎지 않는다.** 대신 다음만 지킨다.

1. **새 기능 상태는 가능하면 전역 static 확산을 줄인다**  
   - 예: 분류 작업은 `FolderDivisionWindow` + 전용 작은 helper 클래스  
2. **파일 쓰기 기능은 항상**  
   - 미리보기 → 확인 → 진행률 → 로그 → 부분 실패 허용  
3. **AppData 설정 vs 폴더 YAML**  
   - 앱 전역: `FastRawSelectorSetting.yaml`  
   - 폴더 선택: `FastRawSelector.yaml`  
4. **네이티브/디코드 변경**  
   - 연속 로드 수동 테스트 + ERROR 로그 확인  

전면 리팩터(async 파이프라인 전면 도입, DI, 테스트 프로젝트)는 **5단계 성능 작업과 맞물릴 때만** 최소 단위로 검토한다.
